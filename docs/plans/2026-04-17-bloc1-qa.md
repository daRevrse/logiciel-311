# Bloc 1 — Admin Mairie : Checklist QA manuelle

Date : 2026-04-18
Contexte : Bloc 1 livré sur `master` (commits de `41a6bf6` à `32dcea7`).

## Pré-requis

- Base MySQL à jour : `cd backend && npx sequelize-cli db:migrate` → 5 migrations appliquées (`20260417100001` → `20260417100005`).
- SMTP configuré (sinon les endpoints `POST /admin/agents` renvoient `temp_password` en dev).
- Au moins 1 mairie, 1 admin, 2 catégories actives.

## 1. Paramètres mairie (D1 → B1/B2)

- [ ] Se connecter en admin, naviguer vers **Ma mairie → Paramètres**.
- [ ] Onglet *Branding* : uploader un logo PNG (> 2 MB → rejet 400), un logo < 2 MB → prévisualisation immédiate.
- [ ] Uploader une bannière > 5 MB → rejet ; < 5 MB → OK.
- [ ] Upload SVG → rejet (allowlist png/jpeg).
- [ ] Modifier `primary_color` et `secondary_color` via color pickers → bloc de preview mis à jour.
- [ ] Onglet *Coordonnées* : remplir address/phone/email + horaires (lundi 09:00-17:00, dimanche fermé) → sauvegarde OK.
- [ ] Onglet *Page publique* : saisir display_name + description + seuil de priorité (min 1). Saisir `0` → rejet côté client ET serveur.
- [ ] Bouton "Enregistrer" → inline success vert ; rechargement page → valeurs persistées.

## 2. Agents (D2 → B3)

- [ ] **Ma mairie → Agents** : page se charge, empty state s'affiche si aucun agent.
- [ ] Bouton "Nouvel agent" → modal :
  - [ ] Email invalide → validation client refuse.
  - [ ] Spécialisations vides → validation client refuse.
  - [ ] Créer avec email valide + 2 catégories → agent apparaît dans la table.
  - [ ] Si SMTP KO (dev) → réponse contient `temp_password` affiché dans un champ copiable + `mail_warning`.
- [ ] Edit un agent existant → changer nom + spécialisations → sauvegarde et refresh.
- [ ] Delete → `window.confirm` → agent passe `is_active=false`.
- [ ] Créer 2 agents avec le même email → premier OK, second → erreur 409 "Email déjà utilisé".

## 3. Interventions + Assignation (D3 → B4/B5)

- [ ] Depuis **Signalements** → ouvrir un report → bouton "Créer intervention".
- [ ] Modal : voir la liste d'agents suggérés. Les agents dont `specializations` inclut la catégorie du report portent le badge **Spécialisé**. Ordre : charge croissante.
- [ ] Sélectionner un agent + date + notes → créer → toast success.
- [ ] Aller dans **Ma mairie → Interventions** : la nouvelle intervention apparaît avec statut `pending`.
- [ ] Vérifier en base `reports.status = 'assigned'` après création (hook `deriveReportStatus`).
- [ ] Cliquer l'intervention → modal détail → changer statut `pending` → `in_progress` → sauver.
- [ ] Vérifier que l'agent reçoit un email d'assignation (ou log mail_service).
- [ ] Vérifier que le citoyen reçoit un email de changement de statut.
- [ ] Filtres : `?status=in_progress`, `?agent_id=X`, `?category_id=Y` → retours filtrés corrects.

## 4. Espace agent (C1/C2/C3 → backend /api/agent)

- [ ] Login avec l'agent (temp password) → redirection automatique vers `/agent`.
- [ ] Page **Mes interventions** : 3 colonnes (À faire / En cours / Terminées), interventions groupées par statut. Les `cancelled` sont cachées.
- [ ] Cliquer une intervention `pending` → page détail → bouton "Démarrer" → passe en `in_progress`, `started_at` rempli.
- [ ] Saisir une note + "Enregistrer" → persistée.
- [ ] Uploader une photo (> 5 MB → rejet ; < 5 MB → OK) → apparait dans la galerie.
- [ ] Bouton "Terminer" → statut `completed`, `completed_at` rempli, report passe `resolved` (si c'est la seule intervention non annulée).
- [ ] Email citoyen envoyé à chaque transition.
- [ ] Tenter de modifier une intervention qui n'est pas la sienne → 404.

## 5. Page publique mairie (E1/E2/E3)

- [ ] Page d'accueil publique → sélecteur mairie → lien "Voir la page de la mairie" → redirection vers `/m/:slug`.
- [ ] Page brandée : logo mairie visible, header `background: var(--primary)` prend la couleur configurée, secondary en accent.
- [ ] Slug invalide → 404 "Mairie introuvable".
- [ ] Description + coordonnées + horaires affichés correctement (jours fermés → "Fermé").
- [ ] CTA "Signaler un problème" → redirection vers le flow de création.
- [ ] Catégories : chips avec icône lucide + couleur.
- [ ] Stats : deux tuiles (résolus / en cours) avec compteurs réels.
- [ ] Recent reports : max 5, exclut les `rejected`.

## 6. Responsive (smoke)

- [ ] Chrome devtools 375px : pages admin settings, agents, interventions restent utilisables (scroll, pas de débordement).
- [ ] 375px : `/agent` + détail intervention utilisables (caméra photo capture OK sur mobile réel).
- [ ] 375px : `/m/:slug` lisible, bannière s'adapte, grille catégories passe à 2 col.

## 7. Sécurité

- [ ] Admin mairie A tente `GET /api/admin/agents` → ne voit que les agents de A.
- [ ] Admin A tente `POST /api/admin/interventions` avec `agent_id` appartenant à mairie B → rejet 400.
- [ ] Admin A tente `DELETE /api/admin/agents/:id_de_B` → 404.
- [ ] Agent connecté à mairie A tente `GET /api/agent/interventions/:id_autre_agent` → 404.
- [ ] Upload logo par super_admin sans `municipality_id` → 400 "Super-admins doivent utiliser /municipalities/:id".
- [ ] `temp_password` NON présent dans la réponse si `NODE_ENV=production`.

## 8. Erreurs gracieuses

- [ ] Upload multer size limit → shape `{ success:false, code, message }`.
- [ ] Notifier mail KO → l'intervention est créée malgré tout (try/catch isolé).
- [ ] Validation express-validator → erreur 400 avec détail.
